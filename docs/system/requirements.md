> **文書ID:** REQ-001
> **対象システム:** Kotonoha — マルチテナント AI チャットボットプラットフォーム
> **作成日:** 2026-03-29
> **ステータス:** 正式版

---

## 1. システム概要

Kotonoha は、組織内のナレッジベースを活用した AI チャットボットプラットフォームである。RAG（Retrieval-Augmented Generation）技術を用いて、登録ドキュメントに基づく高精度な回答を生成し、回答品質の継続的改善を実現する。

### 1.1 システムの目的

- 組織内の問い合わせ対応を AI により自動化し、業務効率を向上させる
- ドキュメントベースの正確な回答により、情報品質を担保する
- 低信頼度回答の自動検出とフィードバックループにより、継続的な品質改善を実現する

### 1.2 対象ユーザー

| ユーザー種別           | 説明                                                                   |
| ---------------------- | ---------------------------------------------------------------------- |
| エンドユーザー         | AI チャットボットを通じて質問・回答を得るユーザー                      |
| 管理者（admin）        | ドキュメント管理、サービス設定、レポート閲覧等の管理業務を行うユーザー |
| 一般メンバー（member） | ログイン済みでチャットおよび自身の会話履歴を参照するユーザー           |
| 外部ユーザー（ゲスト） | Web Component ウィジェット経由でチャットを利用する未認証ユーザー       |

## 4. 業務ルール

| ルールID | ルール名             | 内容                                                                                        |
| -------- | -------------------- | ------------------------------------------------------------------------------------------- |
| BR-01    | 信頼度閾値           | デフォルト 0.6。閾値未満の回答は自動的にフォーム誘導 + 改善リクエスト自動作成               |
| BR-02    | ドキュメント重複     | SHA-256 ハッシュが一致する場合、重複警告を返却。skipDuplicateCheck オプションでスキップ可能 |
| BR-03    | 組織分離             | 全データは organizationId で分離。異なる組織のデータにはアクセス不可                        |
| BR-04    | グループ分離         | 組織内のデータは groupId で更に分離。ユーザーは所属グループのデータのみアクセス可能         |
| BR-05    | 自動登録             | 初回ログイン時にデフォルト組織とグループが自動作成され、ユーザーは admin ロールで登録       |
| BR-06    | チャットメッセージ長 | 最大 10,000 文字                                                                            |
| BR-07    | 会話タイトル         | 最初の質問から自動設定（最大 50 文字）                                                      |
| BR-08    | エスカレーション     | 定義済みキーワード検出時に自動フォーム誘導                                                  |
| BR-09    | レポート保持         | 週次レポートは 90 日間保持                                                                  |
| BR-10    | バッチ処理上限       | Firestore バッチ操作は 490 オペレーション/バッチ                                            |

## 5. 非機能要件

### 5.1 パフォーマンス要件

| 項目                  | 基準値         | 測定条件                                                   | 備考                                                     |
| --------------------- | -------------- | ---------------------------------------------------------- | -------------------------------------------------------- |
| 画面初期表示          | 200ms 以下     | CDN 経由、キャッシュ有効時                                 | SPA（ssr: false）のため初回ロードはやや大きい            |
| チャット応答時間      | 3〜5 秒        | RAG 検索 + Gemini 生成含む                                 | LLM 処理時間が支配的                                     |
| 検索・フィルタ        | 100ms 以下     | Firestore インデックス利用時                               | 複合インデックス事前定義                                 |
| ドキュメント処理      | 非同期         | バックグラウンド処理                                       | ステータス追跡（uploading → processing → ready → error） |
| ベクトル検索          | 500ms 以下     | Firestore Vector Search                                    | 768 次元、FLAT インデックス                              |
| 埋め込み生成          | 1〜2 秒/バッチ | Vertex AI、最大 250 テキスト / 合計 18,000 トークン/バッチ | L2 キャッシュ（Firestore 30 日）で削減                   |
| 外部 API タイムアウト | 30 秒          | Vertex AI / Gemini 呼び出し                                | EXTERNAL_API_TIMEOUT_MS                                  |

## 7. 用語定義

| 用語                       | 定義                                                                                                 |
| -------------------------- | ---------------------------------------------------------------------------------------------------- |
| RAG                        | Retrieval-Augmented Generation。検索で取得した情報をコンテキストとして LLM に渡す回答生成手法        |
| チャンク                   | ドキュメントを分割した単位テキスト。親チャンク（800 トークン）と子チャンク（250 トークン）の階層構造 |
| コンテキストプレフィックス | チャンクの文脈情報を付与する1〜2文のプレフィックス。Contextual Retrieval 技法                        |
| 信頼度スコア               | AI 回答の信頼性を示す 0.0〜1.0 の数値指標                                                            |
| エスカレーション           | AI で解決できない問い合わせを人間対応に引き継ぐこと                                                  |
| フィードバックチャンク     | 管理者が登録した正解回答をベクトル化したデータ。学習ループに使用                                     |
| HyDE                       | Hypothetical Document Embedding。質問から仮想的な回答を生成し、その埋め込みで検索する手法            |
| マルチクエリ               | 1つの質問から複数の検索クエリを生成して検索精度を向上する手法                                        |
